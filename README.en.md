## Cybersecurity Homelab with Wazuh and Metasploitable 3

Documentation of the development and usage of a controlled environment focused on hands-on offensive security practice and defensive monitoring.

The environment integrates:

- Wazuh as a SIEM platform
- Metasploitable 3 VM as a vulnerable target
- Tools executed via WSL2
- VMware Workstation Pro

---

- Practice penetration testing techniques in an isolated environment
- Understand log generation across different services
- Analyse how a SIEM correlates events
- Create custom detection rules
- Simulate a SOC (Security Operations Centre) perspective
- Produce technical documentation

## Environment Deployment

### VMware Network Configuration

In **Virtual Network Editor**:

- **VMnet1** → Host-only → DHCP enabled  
- **VMnet8** → NAT → DHCP enabled  

---

### Wazuh Installation (Ubuntu 24.04)

Create a VM with:

- 6 GB RAM
- 2 CPUs
- 30 GB disk
- Adapter 1: NAT
- Adapter 2: Host-only (VMnet1)

#### Download installer

```bash
curl -sO https://packages.wazuh.com/4.14/wazuh-install.sh
curl -sO https://packages.wazuh.com/4.14/config.yml
sudo bash wazuh-install.sh --generate-config-files
```

#### Cluster configuration (single-node)

Edit `config.yml`:

```yaml
nodes:
  indexer:
    - name: node-1
      ip: "192.168.12.128"
  server:
    - name: wazuh-1
      ip: "192.168.12.128"
  dashboard:
    - name: dashboard
      ip: "192.168.12.128"
```

#### Install components

```bash
sudo bash wazuh-install.sh --wazuh-indexer node-1
sudo bash wazuh-install.sh --start-cluster
sudo bash wazuh-install.sh --wazuh-server wazuh-1
sudo bash wazuh-install.sh --wazuh-dashboard dashboard
```

#### Firewall configuration

```bash
sudo ufw allow 1514/tcp
sudo ufw allow 1514/udp
sudo ufw allow 1515/tcp
sudo ufw allow 55000/tcp
sudo ufw allow 9200/tcp
sudo ufw allow 9300/tcp
sudo ufw allow 443/tcp
sudo ufw allow 5601/tcp
sudo ufw --force enable
```

### Wazuh Agent Installation

```bash

sudo WAZUH_MANAGER="192.168.12.128" apt-get install wazuh-agent
sudo service wazuh-agent start
```

Verification:


## Enumeration with Nmap

From a SOC analyst perspective, it is important to recognise the signals generated by each scan type:

- SYN scan (-sS) → Incomplete connections, many SYN_RECV states
- Version scan (-sV) → Established connections with banner grabbing
- Aggressive scan (-A) → Multiple connections, scripts execution, anomalous patterns
- NSE scripts → Unusual HTTP requests, SMB queries, etc.

```bash
nmap -sV -p1-1000 192.168.12.100
```

- Version scan (-sV) – behaviour:

Establishes full connection with the service

Sends probes to extract banner (version)

##

- Defensive perspective (Logs):

        Apache: access.log with GET / or HEAD /

        SSH: connection attempts without authentication

        FTP: commands STAT, SYST

## Detection with Wazuh

Custom rules file:

```
/var/ossec/etc/rules/local_rules.xml
```

Example rule for SSH brute force:

```xml
<group name="custom,bruteforce">
  <rule id="100100" level="10">
    <if_sid>5716</if_sid>
    <description>Possible SSH brute force detected</description>
  </rule>
</group>
```

Apply rule:

```bash
sudo systemctl restart wazuh-manager
```
